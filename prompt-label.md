下面给你讲一下你要做的事情：

1. rules 是 Rubrics 标注规范，必须严格执行。

2. prompt 是原始需求材料，Rubrics 只能来自 prompt 中明确提出的要求，不要凭空扩写、猜测或补充 prompt 没有要求的内容。

3. 你的任务是把 prompt 拆解成一组可评分的 Rubrics。每条 Rubrics 都应该描述一个具体、独立、可验证的完成要求。

4. 每条 Rubrics 都必须能稳定判断 1 分或 0 分：
   - 满足该条要求可以给 1 分；
   - 未满足、缺失、明显错误或无法验证该条要求可以给 0 分。

5. Rubrics 数量通常控制在 4 到 10 条。若 prompt 要求较复杂，优先覆盖最核心、最影响任务完成度的显式要求。

6. 拆分 Rubrics 时优先覆盖这些内容：
   - 技术框架或运行环境要求，例如 React、浏览器运行、移动端适配等；
   - 核心玩法、核心功能、主要交互和完成目标；
   - 页面或游戏中明确要求展示的信息、状态、按钮、数据或反馈；
   - 明确提出的视觉风格、角色/物体/场景元素、动画和音效；
   - 明确要求的语言、UI 文案或本地化要求。

7. 针对“UI 使用中文”要求：
   - 若整套 Rubrics 条目总数不超过 9 条，将 UI 中文项单独列为一条评分标准；
   - 若无法单独拆分，且已有 UI / 界面设计相关条目，则归入该条目下评分；
   - 若无界面相关条目，则合并至风格类 Rubrics 内评判。

8. 不要写这些类型的 Rubrics：
   - “美观、友好、流畅、现代感”等主观审美描述；
   - “代码结构清晰、模块化、命名规范”等无法从页面稳定判断的工程内部要求；
   - 多个要求混在一条里，导致无法判断到底哪部分得分；
   - 与 prompt 原意不一致的泛化表达。

9. 输出格式必须严格遵守：
   - 只输出编号列表；
   - 每一行一条 Rubrics；
   - 编号格式使用 `1. xxx`；
   - 不要输出标题、解释、总结、Markdown 代码块或任何多余文本。
